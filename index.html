Verifiable location,
by design.

Astral makes computation over location verifiable. When a system answers a question like is this inside that region?, Astral produces a signed, independently checkable record that the answer was computed correctly — without exposing the underlying coordinates.

The problem

Location is easy to claim.
Hard to trust.

Coordinates, check-ins, reported boundaries — all of it can be spoofed, replayed, or quietly faked. And even when the data is honest, the computation on top of it usually isn't checkable: you take the operator's word that the result is what they say it is.

That's fine until the answer carries weight — did this run inside the permitted region, did this stay within the agreed bounds, is this on the right side of the line. Astral is a layer for producing location-based results that someone else can verify, rather than simply trust.

The approach

Two components.

Astral is a small set of components for turning a location-based computation into something a third party can check.

01 / Available now

Verifiable geocompute

Astral runs geospatial operations — distance, area, containment, proximity — and returns the result alongside a signed attestation binding it to its inputs. Anyone can verify the computation was performed correctly, without re-running it. The compute is designed to run inside a trusted execution environment, so the operator can't quietly alter the result.

Signed attestations · TEE-targeted
02 / Working framework

Composable location proofs

A location result is only as credible as the evidence behind it. Astral provides a pipeline for collecting location evidence from independent sources — satellite, network, mobile device attestation, peer witnesses — signing each as a stamp, and bundling them into a single proof with its credibility made explicit. The framework and interface are built; the set of evidence plugins is growing.

Plugin interface live · integrations in progress
Where it applies

Questions where where changes the answer.

The same primitive — a location-based result another party can verify — supports a range of systems, depending on the rule attached to it. A few directions the work points.

AI compute governance

Confirming a workload ran inside a permitted jurisdiction.

A verifiable check against a policy region, without exposing site coordinates or operational detail.

Mobile & sensitive operations

Attesting that operations stayed within agreed bounds.

For field teams, sensor networks, and regulated logistics, at a disclosure level both parties set.

Data sovereignty

Demonstrating residency compliance without continuous monitoring.

A periodic, checkable record that data and processing stayed in a permitted region.

Regulated commitments

After-the-fact proof of where a service was performed.

In sectors where location determines which rules apply, an artifact a reviewer can verify later.

About

A research preview, developed in the open.

Astral is early and openly developed. The protocols, proof constructions, and the open data standard underneath them (the Location Protocol) are published and refined in public — verifiability only means something if the system can be inspected end to end.

The work is developed alongside researchers at the University of Maryland and collaborators across the secure-hardware, cryptography, and geospatial-standards communities.
